## 提交与 PR

- 使用 Conventional Commit，允许 scope，例如 `feat(AuthDropdown): ...`
- 统一使用 `feat`、`fix`、`refactor`、`test`、`style`、`chore`
- 生成 commit 建议时，先看 `git status --short` 的文件状态，再结合变更性质判断类型
- `A`（新增）时，新增业务功能或用户可感知能力优先使用 `feat`；若新增内容仅为测试，使用 `test`；若新增内容仅为配置、脚本、依赖、仓库规范等维护性事项，使用 `chore`
- `M`（修改）时，再根据变更性质判断 `fix`、`refactor`、`style`、`test` 或 `chore`
- `refactor` 仅用于已有文件的结构整理或实现重构，不用于全新文件
- PR 需包含：变更摘要、影响范围、本地验证步骤、UI 变更截图（如适用）
- 修改环境变量、代理、部署行为时必须显式说明

## 全局工作方式

### 禁止指令

禁止使用以下 Git 指令：

- `git restore`
- `git stash`
- `git checkout`
- `git worktree`

### 开发前置要求

- 编写任何代码前，先描述方法，并等待批准
- 如果需求模糊，先提出澄清问题，再开始编码
- 如果任务预计需要修改超过 3 个文件，先暂停并拆分为更小任务
- 如果是在修复 bug，先编写可复现该 bug 的测试，再修复直到测试通过
- 每次被用户纠正时，先反思错误原因，再给出避免复发的措施

### 执行原则

- 不允许在缺乏证据时做假设，结论必须来自现有代码、测试、日志或文档
- 不添加无关文档；用户要的是解决方案时，直接解决问题
- 保持小步交付，确保每次修改后的状态可验证
- 连续三次失败后，必须暂停并重新评估策略

## 语言与输出

- 默认使用简体中文回复；仅在用户明确要求时切换语言
- 最终回复不主动追加“后续可做的增强项”或扩展任务建议
- 输出应适配终端阅读：短段落、清晰分组、必要时使用代码块
- 复杂执行类任务优先使用“任务 / 计划 / 进度 / 风险 / 参考”结构
- 分析类任务优先使用“结论 / 关键分析 / 风险 / 参考”结构

## 调试优先策略

- 不为“先跑起来”而添加静默降级、隐式 fallback、假成功路径或 mock 成功结果
- 不吞掉错误；失败必须以异常、日志、测试失败等方式显式暴露
- 不额外引入边界限制、兜底规则、阻断逻辑，除非是安全要求或用户明确要求
- 如果确实必须增加边界规则或 fallback，必须同时满足：
    - 行为显式
    - 文档可见
    - 容易关闭
    - 事先得到用户同意

## 工程质量基线

### 通用原则

- 遵循 SOLID、DRY、关注点分离、YAGNI
- 命名清晰，抽象务实
- 仅为关键或非直观逻辑添加简洁注释，优先使用简体中文
- 修改行为时，删除无用代码和过时兼容路径；除非用户明确要求保留兼容性
- 关注时间复杂度、空间复杂度、重 I/O 和高内存占用场景
- 显式处理边界条件，不隐藏失败

### 代码硬限制

- 函数长度不超过 50 行（空行不计）
- 单文件不超过 300 行；超过时按职责拆分
- 嵌套深度不超过 3 层；优先使用早返回和 guard clause
- 位置参数不超过 3 个；更多时改为配置对象
- 单函数圈复杂度不超过 10；超出必须拆分
- 不允许 magic number；提取为具名常量

### 解耦与不可变

- 业务逻辑不直接 `new` 具体实现，也不硬绑定具体依赖；优先通过参数或接口注入
- 优先使用不可变数据；不要修改函数入参或全局状态，返回新值

## 安全基线

- 不在源码中硬编码密钥、凭证、Token，统一使用环境变量或密钥管理方案
- 数据库访问必须使用参数化查询，禁止拼接用户输入到 SQL 或命令中
- 在系统边界处验证并清洗外部输入，包括用户输入、API 响应、文件内容
- 用户在对话中粘贴 API Key 属于正常调试流程；仅当密钥被写入源码文件时才需要告警

## 测试与验证

- 代码必须保持可测试，并在可行时使用自动化检查验证
- 能跑测试时就跑测试，优先静态检查、格式化和可复现验证
- 运行后端单元测试时，硬性超时为 60 秒，避免任务卡死
- 完成代码编写后，列出边界情况，并建议覆盖这些边界的测试用例

## 规划与执行

### 规划要求

- 复杂任务在实现前必须先做详尽规划，并记录关键决策
- 跨模块或超过 5 个子任务的工作，必须做任务分解
- 多步骤任务必须维护进度清单，并在执行过程中及时更新
- 计划需要先获得确认，再进入实现

### 执行要求

- 每次工具调用前，明确当前目标与所处步骤
- 先做最小充分的上下文收集，再规划，再执行
- 优先使用只读命令收集上下文，避免大范围无差别扫描
- 仅在验证失败或出现新未知时重新扩展上下文

## 文档与信息唯一性

- 不同层级的文档只描述自己抽象层级内的信息
- 引用其他层资料，不复制粘贴实现细节
- 高层文档不堆叠底层实现细节，保持信息单一来源

## 危险操作确认

以下操作必须先获得用户明确确认：

- 删除文件或目录
- 批量修改文件
- 修改环境变量、系统设置或权限
- 删除数据、改数据库结构、批量更新数据
- 发送敏感数据或调用生产环境 API
- 全局安装、卸载或升级核心依赖

确认时应明确说明：

- 操作类型
- 影响范围
- 风险评估
- 是否继续

## 浏览器自动化

- 需要网页自动化时，优先使用 `agent-browser`
- 基本流程：
    1. `agent-browser open <url>`
    2. `agent-browser snapshot -i`
    3. `agent-browser click @e1` 或 `agent-browser fill @e2 "text"`
    4. 页面变化后重新 `snapshot`

## Shell 与工具约定

- 优先使用只读命令收集信息
- 禁止破坏性命令或强制覆盖，除非用户明确授权
- 大范围操作前先做小范围验证
- 搜索文本或文件时优先使用 `rg` / `rg --files`

## 实施检查清单

- 接手任务后先做现实检查，确认问题和约束
- 在有限上下文内定位到具体待修改文件或函数
- 先有计划，再做修改，再做验证
- 最终交付需要包含关键文件引用、风险说明和验证结果
